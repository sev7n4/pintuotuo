# 决策指导文档

> 本文档提供工作流执行过程中的决策参考，帮助AI做出正确判断。

## 问题类型判断

### 判断流程

```
用户输入
    ↓
包含"bug"、"错误"、"失败"、"异常"？
    ├── 是 → bug
    └── 否 ↓
包含"添加"、"实现"、"新增"、"开发"？
    ├── 是 → feature
    └── 否 ↓
包含"优化"、"重构"、"改进"、"提升"？
    ├── 是 → enhancement
    └── 否 → 询问用户确认
```

### 示例

| 用户输入 | 判断类型 | 理由 |
|----------|----------|------|
| "登录返回401错误" | bug | 包含"错误" |
| "添加商品收藏功能" | feature | 包含"添加" |
| "优化数据库查询性能" | enhancement | 包含"优化" |
| "用户反馈页面加载慢" | bug | 性能问题属于bug |
| "支持多种登录方式" | feature | 新功能 |

## 优先级判断

### 判断标准

| 优先级 | 标准 | 示例 |
|--------|------|------|
| **high** | 影响核心功能、数据安全、用户无法使用 | 登录失败、支付错误、数据丢失 |
| **medium** | 影响次要功能、用户体验下降 | 页面显示异常、搜索结果不准 |
| **low** | 优化改进、非紧急问题 | 代码重构、性能优化 |

### 判断流程

```
问题是否影响用户核心操作？
    ├── 是 → high
    └── 否 ↓
问题是否影响用户体验？
    ├── 是 → medium
    └── 否 → low
```

## 影响范围判断

### 判断标准

| 范围 | 判断依据 |
|------|----------|
| **backend** | 涉及API、数据库、业务逻辑、认证授权 |
| **frontend** | 涉及UI组件、页面交互、样式、前端状态 |
| **both** | 同时涉及前后端 |
| **infra** | 涉及部署、配置、CI/CD |

### 判断流程

```
问题是否涉及API或数据库？
    ├── 是 → 是否同时涉及UI？
    │         ├── 是 → both
    │         └── 否 → backend
    └── 否 ↓
问题是否涉及UI或交互？
    ├── 是 → frontend
    └── 否 → infra
```

## 测试策略决策

### 测试类型选择矩阵

| 影响范围 | 单元测试 | 集成测试 | E2E测试 |
|----------|----------|----------|---------|
| backend | ✅ 必须 | ✅ 必须 | ❌ 不需要 |
| frontend | ✅ 必须 | ❌ 不需要 | ✅ 必须 |
| both | ✅ 必须 | ✅ 必须 | ✅ 必须 |

### TDD流程决策

```
问题类型
    ↓
├── Bug修复
│   └── Red: 写复现测试 → Green: 最小修复 → Refactor: 优化
│
└── 新功能
    └── Red: 写验收测试 → Green: 最小实现 → Refactor: 优化
```

### Bug修复测试策略

```
Bug修复
    ↓
是否涉及API？
    ├── 是 → 编写单元测试 + 集成测试
    └── 否 ↓
是否涉及UI？
    ├── 是 → 编写E2E测试 + 单元测试
    └── 否 → 仅编写单元测试
```

**Bug修复TDD流程**:
```
1. 分析Bug现象，理解预期行为
2. 写单元测试复现Bug (测试应失败)
3. 如果涉及API交互 → 写集成测试
4. 如果涉及用户流程 → 写E2E测试
5. 确认所有测试失败 = Bug被正确捕获
6. 写最小代码修复Bug
7. 确认所有测试通过
8. 添加边界测试用例
9. 重构优化 (可选)
```

### 新功能测试策略

```
新功能
    ↓
影响范围
├── backend → 单元测试 + 集成测试
├── frontend → 单元测试 + E2E测试
└── both → 单元测试 + 集成测试 + E2E测试
```

**新功能TDD流程**:
```
1. 定义验收标准
2. 根据影响范围确定测试类型:
   - both → E2E测试 + 集成测试 + 单元测试
   - backend → 集成测试 + 单元测试
   - frontend → E2E测试 + 单元测试
3. 按顺序编写测试:
   a. E2E测试 (验收测试，定义用户视角的行为)
   b. 集成测试 (API契约测试)
   c. 单元测试 (函数行为定义)
4. 所有测试应失败 (功能未实现)
5. 写最小代码使测试通过
6. 重构优化代码
7. 重复添加更多测试用例
```

### 测试用例设计原则

**单元测试**:
- 测试单个函数/方法的行为
- 使用mock隔离外部依赖
- 覆盖正常路径和边界条件

**集成测试**:
- 测试组件之间的交互
- 使用真实数据库/服务
- 验证API契约

**E2E测试**:
- 测试完整的用户流程
- 模拟真实用户操作
- 验证业务价值

### 测试类型选择

| 场景 | 推荐测试类型 | 原因 |
|------|-------------|------|
| 纯逻辑函数 | 单元测试 | 快速、精确 |
| API端点 | 单元测试 + 集成测试 | 验证接口契约 |
| 数据库操作 | 集成测试 | 需要真实DB环境 |
| 用户流程 | E2E测试 | 验证完整体验 |
| UI组件 | 单元测试 | 隔离测试组件行为 |

### 测试命名决策

```
测试函数命名格式:
Test{FunctionName}_{Scenario}_{ExpectedResult}

场景描述:
- ValidInput / InvalidInput
- EmptyInput / NilInput
- BoundaryCondition
- ConcurrentAccess
- ErrorCondition

示例:
- TestLogin_ValidCredentials_ReturnsToken
- TestLogin_InvalidPassword_ReturnsError
- TestLogin_EmptyEmail_ReturnsValidationError
```

## CI失败处理决策

### 失败类型判断

```
CI失败
    ↓
查看失败日志
    ↓
失败阶段
├── 编译阶段 → 语法错误/依赖问题
├── 单元测试 → 代码逻辑错误
├── 集成测试 → API/数据库问题
├── E2E测试 → UI/流程问题
└── Lint/安全 → 代码规范问题
```

### 处理策略

| 失败类型 | 处理策略 |
|----------|----------|
| 编译错误 | 检查语法、导入、类型 |
| 单元测试失败 | 分析断言，修复代码或测试 |
| 集成测试失败 | 检查数据库连接、API响应 |
| E2E测试失败 | 检查选择器、页面流程 |
| Lint错误 | 按规范修复代码风格 |
| 安全扫描失败 | 检查依赖漏洞，更新版本 |

## 人工介入判断

### 必须人工介入

| 场景 | 原因 |
|------|------|
| 重试5次仍失败 | 可能存在AI无法解决的问题 |
| 影响超过5个文件 | 影响范围过大，需确认 |
| 数据库迁移 | 数据安全风险 |
| 安全敏感操作 | 需要权限确认 |
| 无法定位问题 | 需要用户提供更多信息 |

### 可选人工介入

| 场景 | 建议 |
|------|------|
| 新功能设计 | 可询问用户偏好 |
| 多种解决方案 | 可让用户选择 |
| 性能优化方案 | 可讨论权衡 |

## 分支命名决策

### 类型映射

| 问题类型 | 分支前缀 |
|----------|----------|
| bug | `bugfix/` |
| feature | `feature/` |
| enhancement | `enhancement/` |
| 紧急修复 | `hotfix/` |

### 描述生成

```
从问题标题提取关键词
    ↓
转换为英文（如需要）
    ↓
用连字符连接
    ↓
截断到30字符以内
```

**示例**：
- "登录返回401错误" → `bugfix/issue-001-login-401-error`
- "添加商品收藏功能" → `feature/issue-002-product-favorite`
