# 问题解决计划模板

> **使用说明**：AI根据问题分析结果填充此模板，生成计划文档。

## 基本信息

- **Issue ID**: ISSUE-{三位序号}（如：ISSUE-001）
- **类型**: bug | feature | enhancement
- **优先级**: high | medium | low
- **影响范围**: backend | frontend | both | infra
- **创建时间**: {YYYY-MM-DD HH:mm}

## 问题描述

```
从用户输入提取问题描述，保持原意
```

## 分析结果

### 根本原因
```
分析问题的根本原因（bug）或需求背景（feature）
```

### 影响范围
- **受影响文件**: 列出需要修改的文件路径
- **受影响模块**: 列出涉及的模块名称
- **受影响功能**: 列出受影响的功能点

### 技术方案
```
描述技术解决方案，包括：
- 修改思路
- 关键代码变更点
- 依赖关系变化
```

## 实施计划

### 阶段一：代码分析
- [ ] 搜索相关代码
- [ ] 分析代码依赖
- [ ] 定位问题代码
- [ ] 评估影响范围

### 阶段二：测试设计 (TDD - Red)
- [ ] 设计测试用例
- [ ] 编写失败测试
- [ ] 确认测试失败 (Bug被捕获 / 功能未实现)

**测试用例设计**:
| 测试名称 | 场景 | 预期结果 |
|----------|------|----------|
| Test{Function}_{Scenario} | 输入/条件 | 期望输出 |

### 阶段三：最小实现 (TDD - Green)
- [ ] 实现最小代码
- [ ] 确认测试通过
- [ ] 不添加测试未覆盖的功能

### 阶段四：重构优化 (TDD - Refactor)
- [ ] 消除重复代码
- [ ] 改善代码结构
- [ ] 确认测试仍然通过

### 阶段五：本地验证
- [ ] 运行单元测试
- [ ] 运行集成测试
- [ ] 运行E2E测试
- [ ] 检查代码覆盖率 (Backend ≥85%, Frontend ≥80%)
- [ ] 检查代码风格

### 阶段六：代码提交
- [ ] 提交代码
- [ ] 推送分支
- [ ] 创建PR

### 阶段七：CI验证
- [ ] CI/CD通过
- [ ] 集成测试通过
- [ ] E2E测试通过
