# Anthropic Skills 设计最佳实践案例

> 本文档以 `mydev-github-workflow` SKILL 为例，详细批注每个设计细节如何符合 Anthropic Skills 7大设计原则。

---

## 设计原则总览

| 序号 | 原则 | 说明 |
|------|------|------|
| 1 | 极简设计 | 核心内容在SKILL.md，详细内容在references |
| 2 | 渐进式披露 | 只展示核心流程，需要时展开细节 |
| 3 | 高信号聚焦 | 只保留必要信息，避免噪音 |
| 4 | 清晰触发条件 | description中明确定义何时触发 |
| 5 | 硬约束护栏 | 使用禁止性约束，编号便于引用 |
| 6 | 状态跟踪 | 使用状态文件跟踪进度 |
| 7 | 单一职责 | 每个文件只做一件事 |

### 2. 硬约束区（HARD CONSTRAINTS）

```markdown
## ⚠️ 硬约束 (HARD CONSTRAINTS)

```
✅ Phase 1: PR验证阶段 (PR分支)
   └─ 开发 → 测试 → CI链路验证 → 通过/重试

✅ Phase 2: 合并部署阶段
   └─ 合并 → 部署

[1] 禁止忽略 current_fix_cases 失败！必须修复后才能合并！
[2] 禁止跳过状态跟踪写入！每个步骤必须更新状态文件！
[3] 禁止跳过本地验证！必须通过测试+类型检查才能提交！
[4] 禁止跳过CI监控！必须完成 CI/CD → Integration → E2E 完整链路！
[5] 禁止在CI验证通过前合并PR！必须满足合并条件！
[6] 如果涉及代码逻辑修改，禁止跳过 TDD 流程（Step 5-7）！
```

**重试限制**: 最多 5 次迭代，超过后请求人工介入
```

**📌 设计原则批注**：

| 原则 | 符合度 | 举证说明 |
|------|--------|----------|
| ✅ 原则5: 硬约束护栏 | 100% | 6个约束全部使用禁止性语句，编号[1]-[6]便于流程中引用 |
| ✅ 原则3: 高信号聚焦 | 95% | 约束简洁有力，每个约束一句话说明禁止什么、必须做什么 |
| ✅ 原则1: 极简设计 | 95% | 约束数量适中（6个），不会造成认知负担 |

**💡 最佳实践**：
- 使用"禁止...！必须..."句式，语气强烈
- 每个约束编号，便于在流程步骤中引用
- 约束数量控制在5-7个，过多会造成认知负担
- 先展示Phase概览，再列出具体约束

```markdown
## 工作流链

```
Step 0: 初始化检查
   ↓

   ↓
Step 2-3: 分支创建
   ↓
Step 4: 代码分析
   ↓
Step 5-7: TDD流程 ⚠️ [约束6]
   ↓
Step 8: 本地验证 ⚠️ [约束3]
   ↓
Step 9: git push ──────────────────┐
   ↓                               │ Push 触发
Step 10: CI监控 ⚠️ [约束4] ── CI/CD + Integration
   ├─ 失败 → Step 13 → Step 4/6    │
   └─ 通过                         │
   ↓                               │
Step 11: 创建 PR ──────────────────┤
   ↓                               │ PR 触发
Step 12: E2E监控 ⚠️ [约束1] ── current_fix_cases
   ├─ 失败 → Step 13 → Step 4/6    │
   └─ 通过                         │
   ↓                               │
Step 13: 错误修复 ⚠️ [约束2]        │
   ↓                               │
Step 14: 合并 PR ⚠️ [约束5]         │
   ↓                               │
Step 15: 部署监控 (main分支)        │
   ↓                               │
Step 16: 清理输出                  │
```

**触发逻辑**：
- Push 触发: CI/CD(全量) + Integration(全量)
- PR 触发: E2E(current_fix_cases)

**合并条件**：
- CI/CD Pipeline: 全量通过 ✅
- Integration Tests: 全量通过 ✅
- E2E Tests: current_fix_cases 通过 ✅ (其他用例失败可忽略)
```

| ✅ 原则4: 清晰触发条件 | 100% | 明确标注Push触发和PR触发的位置和方式 |
| ✅ 原则5: 硬约束护栏 | 100% | 关键步骤标注⚠️ [约束X]，引用硬约束 |
| ✅ 原则3: 高信号聚焦 | 95% | 流程图简洁，失败分支清晰，无冗余信息 |

**💡 最佳实践**：
- 使用ASCII流程图，可视化展示Step关系

## 关键设计决策

### 1. 为什么删除"核心流程"章节？

**决策**: 删除与工作流链重复的"核心流程"文字版

**理由**:
- 工作流链可视化更直观
- 避免内容重复
- 符合极简设计原则

### 2. 为什么删除"详细参考"章节？

**决策**: 删除SKILL.md末尾的"详细参考"列表

**理由**:
- 详细参考已在各Step中引用
- 删除可减少文档长度
- 符合高信号聚焦原则

### 3. 为什么约束从9个精简到6个？

**决策**: 合并相似约束，精简数量

**理由**:
- 约束过多会造成认知负担
- 5-7个是最佳数量
- 符合极简设计原则

### 4. 为什么输出格式从18行简化到1行？

**决策**: 简化Step 16输出格式

**理由**:
- 详细输出是噪音
- 核心信息1行足够
- 符合高信号聚焦原则
