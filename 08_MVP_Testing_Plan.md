## 测试策略

### 测试金字塔

```
                    ▲
                   /\
                  /  \  端到端测试 (E2E) (5%)
                 /────\
                /      \  集成测试 (15%)
               /────────\
              /          \  单元测试 (80%)
             /────────────\
            /______________\

覆盖率目标：
  ├─ 单元测试: 单个函数和模块的逻辑 (80%+ 代码覆盖率)
  ├─ 集成测试: 多个模块协同工作 (关键流程100%覆盖)
  └─ E2E测试: 完整用户旅程 (核心场景全覆盖)
```

### 测试类型

```
功能测试:
  ├─ 正常流程测试 (Happy Path)
  ├─ 边界值测试 (Boundary)
  ├─ 异常流程测试 (Error Handling)
  └─ 数据校验测试

性能测试:
  ├─ 接口响应时间 (P99 < 100ms)
  ├─ 并发支持能力 (10,000 QPS)
  ├─ 内存泄漏检查
  └─ 数据库查询效率

安全测试:
  ├─ 认证和授权
  ├─ SQL注入防护
  ├─ XSS防护
  ├─ CSRF防护
  └─ 敏感数据加密

可用性测试:
  ├─ 用户界面易用性
  ├─ 流程清晰度
  ├─ 错误提示友好度
  └─ 移动端适配
```

## 安全测试

### 安全检查清单

```
认证 & 授权:
  [ ] JWT Token签名验证正确
  [ ] Token过期时间设置合理 (7天)
  [ ] 刷新Token机制可用
  [ ] 无效Token被拒绝
  [ ] 权限检查: 用户只能操作自己的数据
  [ ] 商家只能查看自己的订单和数据

数据加密:
  [ ] 密码使用bcrypt加密存储
  [ ] API Key加密存储
  [ ] 敏感字段（身份证号等）使用AES加密
  [ ] 传输层使用HTTPS
  [ ] Cookie使用Secure + HttpOnly标志

SQL注入防护:
  [ ] 所有用户输入都使用参数化查询
  [ ] 没有字符串拼接SQL的代码
  [ ] 测试常见SQL注入payload:
    ├─ ' OR '1'='1
    ├─ '; DROP TABLE users; --
    └─ admin' --

XSS防护:
  [ ] 用户输入被正确转义
  [ ] 前端使用框架的自动转义 (React默认)
  [ ] 测试XSS payload:
    ├─ <script>alert('xss')</script>
    ├─ <img src=x onerror=alert('xss')>
    └─ javascript: alert('xss')

CSRF防护:
  [ ] 敏感操作要求CSRF Token
  [ ] Token在表单中提交

API安全:
  [ ] API限流已实现 (单用户QPS限制)
  [ ] 敏感API要求额外验证 (如修改密码)
  [ ] API文档不暴露内部实现细节
  [ ] 错误信息不泄露系统信息
```

## 缺陷管理

### 缺陷优先级与严重程度

```
优先级 × 严重程度 = 修复时间

优先级:
  P0 (紧急): 当天解决
  P1 (高):   3天内解决
  P2 (中):   1周内解决
  P3 (低):   Post-MVP

严重程度:
  Blocker (阻塞): 影响关键业务逻辑，无workaround
  Critical (严重): 功能不可用或错误，有workaround
  Major (主要): 功能不正确，但不影响使用
  Minor (轻微): 拼写错误、UI问题
  Trivial (微小): 美观问题、文案优化

示例缺陷:
  ┌─────────────────────────────────────────┐
  │ ID: BUG-001                             │
  │ 标题: 拼团支付后余额未更新              │
  │ 优先级: P0 (紧急)                       │
  │ 严重程度: Blocker                       │
  │ 描述: 用户完成支付后，API Key列表中的  │
  │ 余额仍显示旧值，刷新后才更新            │
  │ 预期: 支付完成后立即更新余额            │
  │ 修复: 在支付回调中实时更新Redis缓存     │
  │ 状态: Fixed / Verified                 │
  └─────────────────────────────────────────┘
```

### 缺陷跟踪工具使用

```
使用Jira进行缺陷跟踪:

创建缺陷:
  1. 点击 [Create Issue]
  2. 选择 Issue Type: Bug
  3. 填写:
     ├─ Summary: 简明描述 (50字以内)
     ├─ Description: 详细描述
     ├─ Steps to Reproduce: 复现步骤
     ├─ Expected Result: 预期结果
     ├─ Actual Result: 实际结果
     ├─ Assignee: 分配给开发者
     ├─ Priority: P0-P3
     └─ Labels: 模块标签 (如 "authentication", "payment")

工作流:
  Open → In Progress → In Review → Done
              ↓
            Blocked (如果遇到阻碍)

每日缺陷站会:
  ├─ 汇总新增缺陷数
  ├─ 关注P0缺陷进度
  ├─ 评估发版风险
  └─ 决策: 发版 或 延期修复
```
