## 项目概述

### 项目名称
拼脱脱 (pintuotuo) - 企业Token二级市场平台

### 项目目标（MVP版本）

| 维度 | 目标 |
|------|------|
| **功能** | 完整的C端拼团购买流程 + B端SKU管理 + API调用与计费 |
| **用户** | 首批1000个种子用户，10个B端商家 |
| **业务** | 第一个月GMV突破100万，周均留存率>30% |
| **性能** | API P99延迟 < 100ms, 系统可用性 99.9% |
| **上线** | 第8周完成MVP上线和灰度测试 |

### 项目范围（MVP必做）

**C端功能**：
- ✅ 用户注册、登录、个人中心
- ✅ 首页信息流、商品详情、搜索
- ✅ 拼团购买、订单管理
- ✅ API Key展示、消费明细查询
- ✅ 邀请好友、返利管理

**B端功能**：
- ✅ 商家入驻认证
- ✅ SKU上架与管理
- ✅ API Key托管
- ✅ 实时销售数据看板
- ✅ 月度结算与对账

**平台功能**：
- ✅ 支付网关集成（支付宝、微信）
- ✅ 拼团成团逻辑与自动补贴
- ✅ API路由与计费系统
- ✅ 用户通知（邮件/App推送）
- ✅ 基础监控和告警

### 项目范围外（Post-MVP）

- ❌ 推荐算法（可用热销榜替代）
- ❌ 限时秒杀活动（可用拼团替代）
- ❌ 用户社区和评论系统
- ❌ 复杂的营销工具（满减券等）
- ❌ 企业版定制化功能

## 风险识别与应对

### 风险矩阵

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| 支付网关集成延迟 | 中 | 高 | 提前联系支付宝/微信，预留5天buffer |
| 拼团并发逻辑bug | 高 | 高 | Week 6提前进行并发压测，使用分布式锁 |
| 关键技术人员离职 | 低 | 高 | 代码文档详细，知识共享，团队冗余 |
| 数据库性能问题 | 中 | 中 | Week 3进行早期压力测试，优化索引 |
| 云服务成本超预算 | 中 | 中 | 使用免费额度，定期成本审查 |
| 需求范围蔓延 | 高 | 中 | 严格的需求评审，优先级排序 |

### 应急预案

**场景1：支付集成失败（可能性：中等）**
```
发现时间：Week 5中旬
应对：
  ├─ 立即启动支付宝/微信技术支持
  ├─ 同时实现本地支付Mock（便于前端继续测试）
  ├─ 调整Week 6计划，加强支付集成工作
  └─ 如无法集成，降级为"演示版本"，Post-MVP再正式集成
```

**场景2：性能不达预期（可能性：中等）**
```
发现时间：Week 6压测时
症状：P99延迟 > 200ms 或 无法支持1000 QPS
应对：
  ├─ 分析瓶颈（DB、缓存、网络）
  ├─ 加强缓存策略（Redis）
  ├─ 数据库查询优化 (添加索引)
  ├─ 如仍无法改善，Week 7可考虑削减功能范围
  └─ 灰度阶段严格监控关键指标
```

**场景3：拼团并发bug（可能性：高）**
```
发现时间：Week 7集成测试
应对：
  ├─ 立即停止相关功能测试
  ├─ 专家代码审查（分布式锁实现）
  ├─ 增加单元测试用例（高并发场景）
  ├─ 使用Jmeter等工具进行并发压测验证
  └─ 修复后进行完整回归测试
```

**场景4：核心人员离职（可能性：低）**
```
应对（预防）：
  ├─ 关键代码知识共享 (周Code Review, Architecture讨论)
  ├─ 详细的代码注释和架构文档
  ├─ 新人培养 (Knowledge transfer)

应对（发生时）：
  ├─ 立即安排backup人员接手
  ├─ 简化相关功能范围
  ├─ 可能延期1-2周
```

### 任务跟踪工具

**使用Jira/Linear进行任务管理**

```

  P0 (阻塞): 立即处理，当天解决
  P1 (高):   本周内完成
  P2 (中):   本月内完成
  P3 (低):   Post-MVP

任务状态：
  ToDo → InProgress → InReview → Done

Definition of Done (完成定义)：
  ├─ 代码已提交并通过Code Review
  ├─ 单元测试覆盖率 >= 80%
  ├─ 集成测试通过

```

### 代码管理

```
Git分支策略 (Git Flow)：

  main (生产环线，保护分支)
    ↑
  release (发版分支)
    ↑
  develop (开发分支)
    ↓
  feature/* (功能分支)
    ↓
  bugfix/* (修复分支)

开发流程：
  1. 从develop创建feature/xxx分支
  2. 本地开发，定期commit
  3. Push到远程，发起Pull Request
  4. 代码评审 (至少1人同意)
  5. 测试验证
  6. 合并到develop
  7. 定期从develop合并到release
  8. 发版时，从release合并到main

提交信息规范：
  feat: 新增功能
  fix: 修复bug
  refactor: 代码重构
  test: 单元测试
  docs: 文档更新

  例: "feat: 实现用户注册接口"
```
