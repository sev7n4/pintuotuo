## 一、当前架构分析

### 1.1 现有实现概览

本项目采用了**自研网关架构**，核心实现包括：


| 模块    | 文件                                          | 功能             |
| ----- | ------------------------------------------- | -------------- |
| API代理 | `backend/handlers/api_proxy.go`             | 请求转发、预扣费、响应处理  |
| 智能路由  | `backend/services/smart_router.go`          | 多策略路由、熔断器、评分算法 |
| 健康检查  | `backend/services/health_checker.go`        | 主动/被动健康检查、状态管理 |
| 计费引擎  | `backend/billing/engine.go`                 | 预扣费、返还、成本计算    |
| 模型解析  | `backend/services/openai_compat_resolve.go` | OpenAI兼容格式解析   |


### 1.2 当前架构优势

```
✅ 深度业务集成
   - 与商户系统、SKU/SPU体系、计费引擎无缝集成
   - 预扣费机制与业务逻辑紧密耦合

✅ 灵活的路由策略
   - 支持 price_first、latency_first、balanced、reliability_first
   - 可配置权重和热加载

✅ 完善的健康检查
   - 三级健康检查（high/medium/low）
   - 被动健康检查 + 主动探测
   - 熔断器机制

✅ 多租户架构
   - 商户API密钥隔离
   - 成本独立核算
```

### 1.3 当前架构不足

```
❌ 模型适配层薄弱
   - 仅支持OpenAI兼容格式
   - 缺少对非标准API（如Anthropic原生、Google Gemini）的适配
   - 每新增厂商需手动编写适配代码

❌ 缺少高级路由特性
   - 无Fallback链式降级
   - 无Rate-Limit Aware路由
   - 无跨区域路由

❌ 可观测性不足
   - 缺少分布式追踪（Distributed Tracing）
   - 缺少实时成本看板
   - 缺少告警机制

❌ 运维复杂度高
   - 需自行维护所有适配代码
   - 新模型上线周期长
   - 缺少成熟的社区支持
```

## 二、竞品对标分析

### 2.1 LiteLLM 深度分析

#### 核心架构

```
┌─────────────────────────────────────────────────────────┐
│                    LiteLLM Proxy                        │
├─────────────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │   Router    │  │  Key Manager │  │ Cost Tracker│     │
│  │  (Load Bal) │  │ (Virtual Key)│  │ (Spend)     │     │
│  └─────────────┘  └─────────────┘  └─────────────┘     │
├─────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────┐   │
│  │         Model Adapter Layer (100+ LLMs)         │   │
│  │  OpenAI | Anthropic | Google | Azure | ...      │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

#### 关键特性


| 特性                   | 描述                                                 | 对标当前实现             |
| -------------------- | -------------------------------------------------- | ------------------ |
| **统一接口**             | 100+ LLM统一为OpenAI格式                                | ⚠️ 当前仅支持OpenAI兼容格式 |
| **负载均衡**             | simple-shuffle、least-busy、latency-based、cost-based | ✅ 类似，但LiteLLM更成熟   |
| **虚拟密钥**             | 统一密钥管理，支持预算限制                                      | ⚠️ 当前有商户密钥，但无预算控制  |
| **成本追踪**             | 实时Spend Tracking                                   | ✅ 当前有计费引擎          |
| **Fallback**         | 链式降级机制                                             | ❌ 当前缺失             |
| **Rate-Limit Aware** | 感知配额的路由                                            | ❌ 当前缺失             |
| **性能**               | 1.5k+ req/s                                        | ❓ 未测试              |


#### 配置示例

```yaml
model_list:
  - model_name: gpt-4
    litellm_params:
      model: openai/gpt-4-turbo
      api_key: os.environ/OPENAI_API_KEY
  - model_name: gpt-4
    litellm_params:
      model: azure/gpt-4-turbo
      api_base: https://my-azure.openai.azure.com
      api_key: os.environ/AZURE_API_KEY

router_settings:
  routing_strategy: "latency-based-routing"
  num_retries: 3
  timeout: 30
  allowed_fails: 3
  cooldown_time: 60
```

#### 路由策略详解


| 策略                      | 说明        | 适用场景    |
| ----------------------- | --------- | ------- |
| `simple-shuffle`        | 随机打散（默认）  | 生产环境推荐  |
| `least-busy`            | 最少飞行中请求   | 高并发场景   |
| `latency-based-routing` | 基于延迟      | 响应敏感型应用 |
| `cost-based-routing`    | 基于成本      | 成本敏感型应用 |
| `usage-based-routing`   | 基于TPM/RPM | 配额管理场景  |

### 4.2 理由分析

#### 为什么选择LiteLLM而非OneAPI？


| 因素          | LiteLLM             | OneAPI          | 结论          |
| ----------- | ------------------- | --------------- | ----------- |
| **国际化**     | ✅ 国际主流模型优先          | ⚠️ 国内模型优先       | LiteLLM更适合  |
| **扩展性**     | ✅ Python生态，易扩展      | ⚠️ Go生态，扩展需重新编译 | LiteLLM更灵活  |
| **路由能力**    | ✅ 5种策略+自定义          | ⚠️ 基础轮询         | LiteLLM更强   |
| **社区活跃度**   | ✅ 15k+ GitHub Stars | ⚠️ 5k+ Stars    | LiteLLM更活跃  |
| **企业特性**    | ✅ 虚拟密钥、预算控制         | ✅ 令牌管理、额度控制     | 相当          |
| **与现有架构兼容** | ✅ 可作为SDK嵌入          | ⚠️ 需独立部署        | LiteLLM更易集成 |


#### 为什么不完全替换自研？

```
✅ 核心业务逻辑不可替代
   - 预扣费机制与商户结算深度耦合
   - SKU/SPU体系是产品核心竞争力
   - 多租户成本核算需要定制化

✅ 数据主权要求
   - 用户消费数据、商户账单需本地存储
   - 敏感API密钥不可上传第三方

✅ 差异化竞争
   - 自研计费引擎是产品壁垒
   - 商户管理系统是商业护城河
```

## 五、产品优化建议

### 5.1 借鉴OpenRouter的产品设计

#### 1. 零代码迁移体验

**当前问题**：用户需要修改代码才能接入

```python
# 当前：用户需要修改代码
client = OpenAI(
    base_url="https://api.pintuotuo.com/v1",  # 需要修改
    api_key="pt-xxx"                          # 需要修改
)
```

**优化方案**：提供环境变量一键切换

```bash
# 方案1：环境变量
export OPENAI_API_BASE=https://api.pintuotuo.com/v1
export OPENAI_API_KEY=pt-xxx

# 方案2：配置文件
# ~/.pintuotuo/config.yaml
default:
  base_url: https://api.pintuotuo.com/v1
  api_key: pt-xxx
```

**开发任务**：

- 编写SDK适配器（Python/Node.js）
- 提供环境变量配置支持
- 编写迁移文档

#### 2. 智能模型发现

**当前问题**：用户需要知道具体模型名称

**优化方案**：

```
1. 提供 /v1/models 接口返回可用模型列表
2. 支持模型别名映射（如 gpt-4 → 自动路由到最优提供商）
3. 提供 "auto" 模式，自动选择性价比最优模型
```

**API设计**：

```http
GET /v1/models
Response:
{
  "data": [
```

#### 3. Rate-Limit Aware路由

**当前状态**：无配额感知

**建议实现**：

```go
// 扩展RoutingCandidate结构
type RoutingCandidate struct {
    // ... existing fields
    RPMUsed    int  `json:"rpm_used"`     // 已用RPM
    RPMLimit   int  `json:"rpm_limit"`    // RPM限制
    TPMUsed    int  `json:"tpm_used"`     // 已用TPM
    TPMLimit   int  `json:"tpm_limit"`    // TPM限制
}

// 路由时优先选择配额充足的提供商
func (r *SmartRouter) FilterRateLimited(candidates []RoutingCandidate) []RoutingCandidate {
    var available []RoutingCandidate
    for _, c := range candidates {
        if c.RPMUsed < c.RPMLimit*0.8 && c.TPMUsed < c.TPMLimit*0.8 {
            available = append(available, c)
        }
    }
    return available
}
```

## 六、总结

### 6.1 最终建议


| 时间周期          | 推荐方案                         | 优先级  |
| ------------- | ---------------------------- | ---- |
| **短期（1-2个月）** | 保持自研核心，引入LiteLLM SDK增强模型适配能力 | 🔴 高 |
| **中期（3-6个月）** | 深度集成LiteLLM路由能力，保留业务逻辑层      | 🟡 中 |
| **长期（6个月+）**  | 国内模型在 **LiteLLM** 侧扩展配置；**不**再引入 OneAPI | 🟢 低 |


### 6.2 核心观点

1. **不完全替换**：自研架构在业务集成、多租户、计费方面有独特优势，不应完全放弃
2. **借力生态**：LiteLLM在模型适配、路由策略、社区生态方面成熟度高，可作为底层能力
3. **差异化竞争**：预扣费机制、商户结算、SKU/SPU体系是核心竞争力，应持续优化
4. **渐进式演进**：采用混合架构，逐步引入新能力，降低风险

### 6.3 关键决策点


| 决策点  | 建议选择              | 理由             |
| ---- | ----------------- | -------------- |
| 网关选型 | LiteLLM           | 国际化优先、社区活跃、易集成 |
| 部署方式 | LiteLLM Proxy独立服务 | 解耦清晰、运维简单      |
| 国内模型 | 按需评估              | 先验证市场需求再投入     |
| 迁移策略 | 渐进式               | 降低风险、保留回退能力    |


### 6.4 风险评估


| 风险     | 等级  | 缓解措施        |
| ------ | --- | ----------- |
| 性能下降   | 中   | 充分压测、保留原方案  |
| 学习成本   | 低   | LiteLLM文档完善 |
| 社区依赖   | 低   | 可Fork自行维护   |
| 国内模型缺失 | 中   | 在 LiteLLM 增加 dashscope/zai 等路由与 Key |
