## 二、什么是 Harness Engineering？

### 2.1 核心定义

> **Harness Engineering（驾驭工程）** 是设计和实现约束、护栏、反馈循环和生命周期工具的工程实践，其目的是让 AI Agent 持续产生正确、可审计、可维护的输出。

### 2.2 生动类比：马与马具

想象这样一个场景：

```
┌─────────────────────────────────────────────────────────────┐
│                      🐴 骏马 (LLM)                          │
│                                                             │
│   能力：跑得快、力量大、聪明                                 │
│   问题：没有方向感、容易受惊、难以控制                       │
│                                                             │
│   在空旷草原上跑得很快很好看，但完全没法干活                 │
└─────────────────────────────────────────────────────────────┘
                            ⬇️ 套上马具
┌─────────────────────────────────────────────────────────────┐
│                   🐴➕🏇 驯服的战马                          │
│                                                             │
│   缰绳 → 控制方向（Context Engineering）                     │
│   马鞍 → 稳定骑乘（Architectural Constraints）              │
│   围栏 → 防止乱跑（Entropy Management）                      │
│   跑道 → 明确路径（Workflow Control）                        │
│                                                             │
│   现在可以拉车、耕地、送信——真正干活了！                     │
└─────────────────────────────────────────────────────────────┘
```

**生活实例**：
- **没有 Harness**：就像让一个天才厨师在没有厨房、没有食谱、没有食材清单的情况下做菜——他可能做出美味，也可能把厨房烧了
- **有了 Harness**：提供标准化的厨房设备、清晰的食谱流程、质量检查清单——每次都能稳定产出美味佳肴

### 2.3 为什么需要 Harness？

**根本原因：LLM 是无状态的！**

```
┌─────────────────────────────────────────────────────────────┐
│                    LLM 的本质局限                            │
├─────────────────────────────────────────────────────────────┤
│ ❌ 无记忆：调一次 API 返回一个结果就结束了                   │
│ ❌ 不会自己调用自己                                          │
│ ❌ 不会记住上一轮说了什么                                    │
│ ❌ 输出不确定：同样的输入可能得到不同的输出                  │
└─────────────────────────────────────────────────────────────┘
```

**生活实例**：
想象你雇了一个超级聪明但有以下特点的助手：
- 每次对话后都会"失忆"
- 你必须把所有背景信息重新说一遍
- 他不会主动追问或确认
- 同样的任务可能给出完全不同的方案

这就是为什么需要 Harness——它就像一个"外置大脑"，帮 LLM 记住、规划、执行、检查。

## 三、三大核心支柱详解

根据 Martin Fowler 的归纳，Harness Engineering 包含三大支柱：

### 3.1 支柱一：Context Engineering（上下文工程）

**核心问题**：Agent 应该看到什么？

```
┌─────────────────────────────────────────────────────────────┐
│                  Context Engineering                         │
│                                                             │
│   从 Agent 的视角看，任何它在上下文中无法访问的东西都不存在  │
│                                                             │
│   Slack 里的讨论 ❌ 不可见                                   │
│   人脑中的知识 ❌ 不可见                                     │
│   未提交的代码 ❌ 不可见                                     │
│                                                             │
│   必须把所有 Agent 需要知道的东西编码进代码仓库：            │
│   ✅ 架构决策                                                │
│   ✅ 设计文档                                                │
│   ✅ 质量标准                                                │
│   ✅ 执行计划                                                │
│   全部版本化、可发现                                         │
└─────────────────────────────────────────────────────────────┘
```

**生活实例**：

想象你是一个新入职的员工，公司没有给你任何文档：
- 不知道公司有哪些系统
- 不知道代码规范是什么
- 不知道找谁问问题
- 不知道历史决策原因

**Context Engineering 就是给 Agent 准备一份"新员工入职手册"**，包括：
- 公司架构图（系统架构）
- 工作流程手册（开发规范）
- 历史决策记录（ADR 文档）
- 常见问题解答（FAQ）

**OpenAI 的实践**：
- 一个约 100 行的 `AGENTS.md` 作为"地图"注入上下文，指向更深层的文档
- **关键发现**：一个巨大的 AGENTS.md 反而会失效——上下文是稀缺资源，塞太多 Agent 反而抓不住重点

**代码示例**：

```python
# AGENTS.md 示例结构
"""
# 项目概览
这是一个电商系统，包含用户、订单、支付模块

# 架构决策
- 使用 PostgreSQL 作为主数据库
- Redis 用于缓存和会话
- 所有 API 必须通过 /api/v1/ 前缀

# 代码规范
- 使用 TypeScript
- 测试覆盖率 > 80%
- 提交信息遵循 Conventional Commits

# 常见任务
- 添加新 API: 参考 /docs/api-guide.md
- 修改数据库: 参考 /docs/db-migration.md
"""
```

### 3.2 支柱二：Architectural Constraints（架构约束）

**核心问题**：系统应该防止什么？

```
┌─────────────────────────────────────────────────────────────┐
│              Architectural Constraints                       │
│                                                             │
│   光在文档里写规则，Agent 仍然可能违反                       │
│   在系统层面强制执行，则从根本上防止违规                      │
│                                                             │
│   由自定义 Linter 和结构测试强制执行的确定性规则             │
└─────────────────────────────────────────────────────────────┘
```

**生活实例**：

想象一个工厂：

| 方式 | 效果 |
|------|------|
| 在墙上贴标语"请戴安全帽" | ❌ 工人可能忘记 |
| 在入口设置感应门，不戴安全帽无法通过 | ✅ 强制执行 |

**OpenAI 的实践**：
- 静态强制了结构化日志
- 命名规范
- 文件大小限制
- 依赖方向：`Types → Config → Repo → Service → Runtime → UI`（不可逆向）

**关键细节**：
- Linter 报错时，错误信息被设计为直接注入 Agent 上下文
- 再将错误信息传给 LLM，形成自动修复的反馈循环

```
┌─────────────────────────────────────────────────────────────┐
│                    自动修复反馈循环                          │
│                                                             │
│   Agent 写代码 → Linter 检查 → 发现错误                      │
│        ⬆️                           │                        │
│        │                           ⬇️                        │
│   Agent 修复代码 ← 注入错误信息 ← 格式化错误                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**代码示例**：

```yaml
# 自定义 Linter 规则示例
rules:
  - name: no-circular-dependency
    description: 禁止循环依赖
    severity: error
    
  - name: file-size-limit
    description: 单文件不超过 500 行
    config:
      maxLines: 500
    severity: warning
    
  - name: dependency-direction
    description: 依赖方向检查
    allowed:
      - Types → Config
      - Config → Repo
      - Repo → Service
      - Service → Runtime
      - Runtime → UI
    severity: error
```

### 3.3 支柱三：Entropy Management（熵管理）

**核心问题**：如何持续维护质量？

```
┌─────────────────────────────────────────────────────────────┐
│                   Entropy Management                         │
│                                                             │
│   本质：代码质量的"垃圾回收"                                 │
│                                                             │
│   人类品味捕获一次，持续自动执行                              │
│                                                             │
│   Agent 定期运行，清理文档不一致、违反架构约束的内容          │
│   对抗系统衰退                                               │
└─────────────────────────────────────────────────────────────┘
```

**生活实例**：

想象你的房间：

| 情况 | 结果 |
|------|------|
| 从不打扫 | 房间越来越乱，最终无法居住 |
| 每周手动打扫 | 需要花大量时间，容易忘记 |
| 机器人自动打扫 | 每天保持整洁，省心省力 |

**OpenAI 的实践**：
- 过去每周五花一天时间手动清理 Agent 生成的低质量代码（"AI slop"）
- 后来自动化了：Agent 扫描偏差、提交小的重构 PR、大部分自动合并

**熵管理的层次**：

```
┌─────────────────────────────────────────────────────────────┐
│                    熵管理金字塔                              │
│                                                             │
│                    ┌─────────┐                              │
│                    │ 人工审核 │ ← 最顶层，处理复杂问题        │
│                   ┌┴─────────┴┐                             │
│                   │ 自动 PR   │ ← 中层，自动修复常见问题      │
│                  ┌┴───────────┴┐                            │
│                  │ Linter/测试 │ ← 基础层，实时检查           │
│                 └─────────────┘                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 四、Harness 的技术架构

### 4.1 核心组件

```
┌─────────────────────────────────────────────────────────────┐
│                    Harness 核心架构                          │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                   Agent Loop                         │   │
│  │  ┌─────────┐    ┌─────────┐    ┌─────────┐         │   │
│  │  │ 输入处理 │ → │ LLM推理  │ → │ 输出路由 │         │   │
│  │  └─────────┘    └─────────┘    └─────────┘         │   │
│  │       ⬆️                             │               │   │
│  │       │                             ⬇️               │   │
│  │  ┌─────────┐    ┌─────────┐    ┌─────────┐         │   │
│  │  │ 上下文  │ ← │ 工具执行 │ ← │ 结果处理 │         │   │
│  │  │ 管理    │    │         │    │         │         │   │
│  │  └─────────┘    └─────────┘    └─────────┘         │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                   护栏系统                           │   │
│  │  • 最大步数限制    • 超时控制                        │   │
│  │  • 错误重试机制    • 输出验证                        │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                   可观测性                           │   │
│  │  • 日志记录    • 追踪信息    • 性能指标              │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 简单 Harness 实现

```python
class SimpleHarness:
    def __init__(self, llm, tools, max_steps=10):
        self.llm = llm
        self.tools = tools
        self.max_steps = max_steps  # Architectural Constraint
    
    def run(self, user_query):
        messages = [
            {"role": "system", "content": self._build_system_prompt()},
            {"role": "user", "content": user_query}
        ]
        
        for step in range(self.max_steps):  # Workflow Control
            response = self.llm.chat(messages)    # Context Engineering
            parsed = self._parse_response(response)
            
            if parsed["type"] == "final_answer":
                return parsed["content"]
            elif parsed["type"] == "tool_call":
                tool_result = self._execute_tool(
                    parsed["tool"], 
                    parsed["args"]
                )
                messages.append({"role": "assistant", "content": response})
                messages.append({"role": "tool", "content": tool_result})
        
        return "达到最大步数。"  # Architectural Constraint
```

**代码解读**：

| 组件 | 对应概念 | 作用 |
|------|----------|------|
| `messages` 列表 | Context Engineering | 管理 Agent 能看到什么 |
| `for` 循环 | Workflow Control | 控制执行流程 |
| `max_steps` | Architectural Constraint | 防止无限循环 |
| `_execute_tool` | 工具调用 | 扩展 Agent 能力 |

### 4.3 消息角色体系

```
┌─────────────────────────────────────────────────────────────┐
│                    消息角色体系                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  system    系统提示词                                        │
│     │     Agent 的身份、规则和能力边界                       │
│     │     对应 AGENTS.md + 系统指令                          │
│     ⬇️                                                       │
│  user      用户输入                                          │
│     │     模型会被训练为回应 user 的请求                     │
│     │     Agent 就是为了处理用户的输入并做出正确行为          │
│     ⬇️                                                       │
│  assistant 模型输出                                          │
│     │     Harness 会将其放回消息列表                         │
│     │     这样模型才会"记住"他之前说了啥，做了啥             │
│     ⬇️                                                       │
│  tool      工具执行结果                                      │
│           Harness 将工具执行结果回传给模型                   │
│           模型根据此再继续进行推理                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.4 上下文管理策略

由于模型的上下文是有限的，需要管理策略：

```
┌─────────────────────────────────────────────────────────────┐
│                  上下文管理策略                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 截断/摘要                                               │
│     工具输出过大时先压缩                                     │
│     例：读取技术方案，先用小模型摘要                         │
│                                                             │
│  2. 滑动窗口                                                │
│     只保留最近 N 轮循环的上下文                              │
│                                                             │
│  3. 按需注入                                                │
│     维护外部记忆，每次只注入当前步骤需要的                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Claude Code 的四层上下文管理**：

```
源头控制 → 工具结果清理 → 自动 Compaction → Subagent 分流
```

## 五、Mitchell Hashimoto 的 AI 采用之旅

Mitchell Hashimoto（HashiCorp 联合创始人）分享了他的六步 AI 采用之旅：

### 5.1 Step 1: 放弃聊天机器人

```
┌─────────────────────────────────────────────────────────────┐
│                    Step 1: Drop the Chatbot                  │
│                                                             │
│  ❌ 不要通过聊天界面（ChatGPT、Gemini 等）进行有意义的编码   │
│                                                             │
│  原因：                                                      │
│  • 你主要希望它们基于训练数据给出正确结果                    │
│  • 纠正它们需要人类反复告诉它们错了                          │
│  • 效率很低                                                  │
│                                                             │
│  ✅ 必须使用 Agent                                          │
│  Agent = LLM + 聊天 + 外部行为调用的循环                     │
│  最小能力：读文件、执行程序、发 HTTP 请求                    │
└─────────────────────────────────────────────────────────────┘
```

**生活实例**：
- 聊天机器人就像电话客服——你描述问题，他给建议，但不亲自解决
- Agent 就像上门维修工——亲自查看、诊断、修复

### 5.2 Step 2: 重现自己的工作

```
┌─────────────────────────────────────────────────────────────┐
│               Step 2: Reproduce Your Own Work                │
│                                                             │
│  方法：                                                      │
│  1. 手动完成工作                                             │
│  2. 让 Agent 在看不到你的解决方案的情况下重现相同结果         │
│  3. 真正做两遍工作                                           │
│                                                             │
│  学到的经验：                                                │
│  • 将会话分解为清晰、可执行的任务                            │
│  • 不要试图在一个大会话中"画猫头鹰"                         │
│  • 模糊请求：拆分为规划会话 + 执行会话                       │
│  • 给 Agent 验证工作的方法，它会自己修复错误                 │
└─────────────────────────────────────────────────────────────┘
```

**"画猫头鹰"问题**：

```
如何画猫头鹰：

Step 1: 画两个圆
Step 2: 画剩下的猫头鹰

❌ 这就是模糊指令的问题
```

### 5.3 Step 3: 每日结束时的 Agent

```
┌─────────────────────────────────────────────────────────────┐
│               Step 3: End-of-Day Agents                      │
│                                                             │
│  方法：每天最后 30 分钟启动一个或多个 Agent                  │
│                                                             │
│  适合的任务：                                                │
│  • 深度研究：调研某个领域，生成多页摘要                      │
│  • 并行尝试：多个 Agent 尝试不同的模糊想法                   │
│  • Issue/PR 分类：用 gh CLI 自动分类                        │
│                                                             │
│  好处：第二天早上有"热启动"，比平时更快开始工作              │
└─────────────────────────────────────────────────────────────┘
```

### 5.4 Step 4: 外包确定能做的事

```
┌─────────────────────────────────────────────────────────────┐
│             Step 4: Outsource the Slam Dunks                 │
│                                                             │
│  方法：                                                      │
│  1. 从昨晚的分类结果中筛选 Agent 几乎肯定能解决好的问题      │
│  2. 让 Agent 在后台处理（一次一个，不并行）                  │
│  3. 你同时做其他事情                                        │
│                                                             │
│  重要：关闭 Agent 桌面通知                                   │
│  上下文切换很昂贵，由你控制何时检查 Agent                    │
└─────────────────────────────────────────────────────────────┘
```

### 5.5 Step 5: 工程化 Harness

```
┌─────────────────────────────────────────────────────────────┐
│             Step 5: Engineer the Harness                     │
│                                                             │
│  核心思想：                                                  │
│  每当你发现 Agent 犯了一个错误，                             │
│  你花时间设计一个解决方案，                                  │
│  使 Agent 永远不会再犯那个错误。                             │
│                                                             │
│  两种形式：                                                  │
│                                                             │
│  1. 更好的隐式提示（AGENTS.md）                              │
│     例：Agent 反复运行错误命令                               │
│     解决：更新 AGENTS.md 说明正确命令                        │
│                                                             │
│  2. 实际编程的工具                                           │
│     例：截图脚本、过滤测试脚本                               │
│     通常配合 AGENTS.md 更新让 Agent 知道这些工具存在         │
└─────────────────────────────────────────────────────────────┘
```

**Ghostty 项目 AGENTS.md 示例**：
- 每一行都是基于 Agent 的不良行为添加的
- 几乎完全解决了所有这些问题

### 5.6 Step 6: 始终让 Agent 运行

```
┌─────────────────────────────────────────────────────────────┐
│           Step 6: Always Have an Agent Running               │
│                                                             │
│  目标：始终有一个 Agent 在运行                               │
│                                                             │
│  如果没有 Agent 在运行，问自己：                             │
│  "现在有什么 Agent 可以为我做的吗？"                         │
│                                                             │
│  实践：                                                      │
│  • 目前大约 10-20% 的工作时间有后台 Agent 在运行             │
│  • 正在努力提高这个比例                                      │
│  • 只在真正有帮助的任务上运行 Agent                          │
└─────────────────────────────────────────────────────────────┘
```

## 六、Anthropic Agent 构建最佳实践

### 6.1 何时使用（何时不使用）Agent

```
┌─────────────────────────────────────────────────────────────┐
│                  何时使用 Agent                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  推荐原则：                                                  │
│  寻找最简单的解决方案，只在需要时增加复杂度                  │
│                                                             │
│  复杂度递增：                                                │
│                                                             │
│  1. 单次 LLM 调用 + 检索 + 上下文示例                        │
│     ↓ 大多数应用这就够了                                     │
│                                                             │
│  2. Workflow（工作流）                                       │
│     预定义代码路径编排 LLM 和工具                            │
│     适合：定义明确的任务，需要可预测性和一致性                │
│                                                             │
│  3. Agent（智能体）                                          │
│     LLM 动态控制自己的流程和工具使用                         │
│     适合：需要灵活性和模型驱动决策的大规模场景                │
│                                                             │
│  权衡：Agent 通常用延迟和成本换取更好的任务性能              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 五种常见工作流模式

## 七、Anthropic Agent 评估最佳实践

### 7.1 为什么需要评估？

```
┌─────────────────────────────────────────────────────────────┐
│                   为什么需要评估？                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  没有评估的困境：                                            │
│  ❌ 被动调试：等用户投诉 → 手动复现 → 修复 → 希望没别的回归  │
│  ❌ 无法区分真实回归和噪音                                   │
│  ❌ 无法在发布前自动测试数百个场景                           │
│  ❌ 无法衡量改进效果                                         │
│                                                             │
│  有评估的好处：                                              │
│  ✅ 问题在影响用户前被发现                                   │
│  ✅ 区分真实回归和随机噪音                                   │
│  ✅ 自动测试变更                                             │
│  ✅ 衡量改进                                                 │
│  ✅ 价值随时间复合                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 评估结构

```
┌─────────────────────────────────────────────────────────────┐
│                     评估结构                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  任务（Task）                                                │
│  • 单个测试，有定义的输入和成功标准                          │
│                                                             │
│  试验（Trial）                                               │
│  • 对任务的一次尝试                                          │
│  • 因为模型输出有变化，需要多次试验                          │
│                                                             │
│  评分器（Grader）                                            │
│  • 对 Agent 某方面表现评分的逻辑                             │
│  • 一个任务可以有多个评分器                                  │
│                                                             │
│  记录（Transcript）                                          │
│  • 试验的完整记录：输出、工具调用、推理、中间结果            │
│                                                             │
│  结果（Outcome）                                             │
│  • 试验结束时环境的最终状态                                  │
│                                                             │
│  评估 Harness（Evaluation Harness）                          │
│  • 端到端运行评估的基础设施                                  │
│                                                             │
│  Agent Harness                                               │
│  • 让模型作为 Agent 运行的系统                               │
│  • 评估"一个 Agent"就是评估 Harness + 模型                   │
│                                                             │
│  评估套件（Evaluation Suite）                                │
│  • 设计用于测量特定能力或行为的任务集合                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.3 三种评分器类型

```
┌─────────────────────────────────────────────────────────────┐
│                    三种评分器类型                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 代码评分器 (Code-based Graders)                      │   │
│  │                                                     │   │
│  │ 方法：字符串匹配、二进制测试、静态分析、结果验证      │   │
│  │ 优点：快速、便宜、客观、可复现、易调试               │   │
│  │ 缺点：对有效变体脆弱、缺乏细微差别                   │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 模型评分器 (Model-based Graders)                     │   │
│  │                                                     │   │
│  │ 方法：基于评分标准、自然语言断言、成对比较           │   │
│  │ 优点：灵活、可扩展、捕获细微差别、处理开放任务       │   │
│  │ 缺点：非确定性、比代码评分贵、需要人工校准           │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 人工评分器 (Human Graders)                           │   │
│  │                                                     │   │
│  │ 方法：专家审查、众包判断、抽查采样、A/B 测试         │   │
│  │ 优点：黄金标准质量、匹配专家判断                     │   │
│  │ 缺点：昂贵、慢、需要大规模专家资源                   │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.4 能力评估 vs 回归评估

```
┌─────────────────────────────────────────────────────────────┐
│              能力评估 vs 回归评估                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  能力评估（Capability Evals）                                │
│  问题："这个 Agent 能做好什么？"                            │
│  特点：从低通过率开始，给团队一个攀登的目标                  │
│                                                             │
│  回归评估（Regression Evals）                                │
│  问题："Agent 还能处理以前能做的所有任务吗？"               │
│  特点：应该有接近 100% 的通过率                             │
│       防止倒退，分数下降说明有问题                          │
│                                                             │
│  关系：                                                      │
│  • 攀登能力评估时，也要运行回归评估                          │
│  • 能力评估高通过率后可以"毕业"成为回归评估                 │
│  • "我们能做到吗？" → "我们还能可靠地做到吗？"              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.6 处理非确定性

```
┌─────────────────────────────────────────────────────────────┐
│                   处理非确定性                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  问题：Agent 行为在运行间有变化                              │
│                                                             │
│  两个关键指标：                                              │
│                                                             │
│  pass@k：k 次尝试中至少一次正确的概率                       │
│  • k 增加时，pass@k 上升                                    │
│  • 例：50% pass@1 = 第一次尝试成功一半任务                  │
│  • 编码中通常关心 pass@1（第一次就找到解决方案）            │
│                                                             │
│  pass^k：k 次试验全部成功的概率                             │
│  • k 增加时，pass^k 下降                                    │
│  • 例：75% 单次成功率，3 次试验全通过概率 = 0.75³ ≈ 42%     │
│  • 对面向用户的 Agent 很重要，用户期望每次都可靠            │
│                                                             │
│  选择：                                                      │
│  • pass@k：工具类，一次成功就够                             │
│  • pass^k：Agent 类，一致性至关重要                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.7 从零到一的路线图

```
┌─────────────────────────────────────────────────────────────┐
│                评估建设路线图                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 0: 尽早开始                                            │
│  • 20-50 个简单任务就是很好的开始                           │
│  • 等得越久，评估越难构建                                    │
│                                                             │
│  Step 1: 从手动测试开始                                      │
│  • 转换已报告的失败为测试用例                                │
│  • 按用户影响优先排序                                        │
│                                                             │
│  Step 2: 编写无歧义的任务                                    │
│  • 两个领域专家应独立得出相同的通过/失败结论                 │
│  • 创建参考解决方案验证任务可解                              │
│                                                             │
│  Step 3: 构建平衡的问题集                                    │
│  • 测试行为应该发生和不应该发生的情况                        │
│  • 避免类别不平衡                                            │
│                                                             │
│  Step 4: 构建稳健的评估 Harness                              │
│  • 每次试验从干净环境开始                                    │
│  • 避免共享状态导致的关联失败                                │
│                                                             │
│  Step 5: 深思熟虑地设计评分器                                │
│  • 尽可能使用确定性评分器                                    │
│  • 必要时使用 LLM 评分器                                     │
│  • 谨慎使用人工评分器                                        │
│                                                             │
│  Step 6: 检查记录                                            │
│  • 阅读失败案例的记录                                        │
│  • 验证评分器是否公平                                        │
│                                                             │
│  Step 7: 监控能力评估饱和                                    │
│  • 100% 的评估只追踪回归，不提供改进信号                    │
│                                                             │
│  Step 8: 长期维护                                            │
│  • 评估套件是需要持续关注的活文档                            │
│  • 让最接近产品需求的人定义成功                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 8.2 工具设计实践

```
┌─────────────────────────────────────────────────────────────┐
│                   工具设计实践                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 站在模型角度思考                                         │
│     • 基于描述和参数，工具用法是否明显？                    │
│     • 好的工具定义包括：示例用法、边缘情况、输入格式要求    │
│                                                             │
│  2. 优化参数名和描述                                         │
│     • 就像为团队初级开发者写优秀的文档字符串                │
│                                                             │
│  3. 测试模型如何使用工具                                     │
│     • 运行多个示例输入，观察模型犯什么错误                  │
│     • 迭代改进                                               │
│                                                             │
│  4. 防错设计（Poka-yoke）                                    │
│     • 改变参数使更难犯错                                    │
│                                                             │
│  实例：                                                      │
│  • 发现模型在离开根目录后使用相对路径会出错                 │
│  • 解决：改为始终要求绝对路径                               │
│  • 结果：模型完美使用该方法                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 十、总结与行动清单

### 10.1 核心要点总结

```
┌─────────────────────────────────────────────────────────────┐
│                     核心要点                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. Harness 是"确定性的壳"包裹"不确定性的核"                │
│     LLM 输出不确定，Harness 用确定性代码约束它               │
│                                                             │
│  2. Harness 拥有状态，LLM 没有                               │
│     通过 Harness 维护消息列表、Trace、执行步数               │
│                                                             │
│  3. Harness 定义 Agent 的"能力边界"                          │
│     能调哪些工具、最多循环几次、错误怎么处理                 │
│                                                             │
│  4. 模型质量在快速趋同，Harness 是长期竞争力                 │
│     每个团队必须自己搭建的资产                               │
│                                                             │
│  5. 评估是不可或缺的基础设施                                 │
│     没有评估，就是在黑暗中摸索                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
