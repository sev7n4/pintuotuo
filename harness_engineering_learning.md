## 第一章：本质理解——为什么需要 Harness？

### 1.1 核心定义

> **Harness Engineering（驾驭工程）** 是设计和实现约束、护栏、反馈循环和生命周期工具的工程实践，其目的是让 AI Agent 持续产生正确、可审计、可维护的输出。

### 1.2 根本问题：LLM 的"不确定性本质"

要理解 Harness，首先要理解 LLM 的本质局限。

#### 🧠 认知模型：LLM 是"概率预测机"

```
┌─────────────────────────────────────────────────────────────┐
│              LLM 的本质：概率预测机                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   输入: "今天天气"                                           │
│                                                             │
│   输出概率分布:                                              │
│   ┌────────────────────────────────────────────┐            │
│   │ "很好"     ████████████████████  35%       │            │
│   │ "不错"     ██████████████      25%         │            │
│   │ "晴朗"     ████████████        22%         │            │
│   │ "糟糕"     ████████            15%         │            │
│   │ "下雨了"   ████                8%          │            │
│   └────────────────────────────────────────────┘            │
│                                                             │
│   每次调用，LLM 从概率分布中"采样"一个结果                   │
│   这就是为什么同样输入可能得到不同输出                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**生活实例**：想象一个超级聪明但"不靠谱"的天才朋友：

| 特点 | 表现 | 类比 |
|------|------|------|
| 无记忆 | 每次对话后"失忆" | 像《记忆碎片》的主角，每次醒来都不记得之前的事 |
| 无状态 | 不会记住上一轮说了什么 | 像打电话每次都重新拨号，对方不记得你是谁 |
| 概率输出 | 同样问题可能给不同答案 | 像掷骰子选答案，虽然骰子偏向正确答案 |
| 无自我纠错 | 不会主动检查自己 | 像考试交卷前不检查的学生 |

#### 🎯 工程哲学：确定性与不确定性的博弈

```
┌─────────────────────────────────────────────────────────────┐
│                    工程哲学核心                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   软件工程的本质追求：确定性                                  │
│   ─────────────────────────────────────                     │
│   • 同样的输入 → 同样的输出                                  │
│   • 可预测、可复现、可调试                                   │
│   • 例：1 + 1 永远等于 2                                    │
│                                                             │
│   LLM 的本质属性：不确定性                                    │
│   ─────────────────────────────────────                     │
│   • 同样的输入 → 可能不同的输出                              │
│   • 概率性、随机性、创造性                                   │
│   • 例："写一首诗" 每次都不同                               │
│                                                             │
│   Harness Engineering 的哲学：                               │
│   ─────────────────────────────────────                     │
│   用确定性的"壳"包裹不确定性的"核"                           │
│   让 AI 的创造力在安全边界内发挥                             │
│                                                             │
│   ┌─────────────────────────────────────┐                   │
│   │     确定性的 Harness（外壳）         │                   │
│   │  ┌─────────────────────────────┐    │                   │
│   │  │   不确定性的 LLM（内核）     │    │                   │
│   │  │   创造力 + 智能              │    │                   │
│   │  └─────────────────────────────┘    │                   │
│   │  约束 + 护栏 + 反馈 + 验证          │                   │
│   └─────────────────────────────────────┘                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**生活实例**：

| 场景 | 没有 Harness | 有 Harness |
|------|-------------|------------|
| **赛车** | 车手想怎么开就怎么开，可能冲出赛道 | 有赛道边界、安全护栏、规则限制，车手在边界内尽情发挥 |
| **河流** | 洪水泛滥，方向不可控 | 修筑堤坝、水闸，让水流在可控范围内发电灌溉 |
| **孩子玩耍** | 在马路上乱跑，危险 | 在游乐场围栏内玩耍，安全又自由 |
| **AI 编程** | 可能写出有 bug 的代码 | 有测试、Linter、代码审查，确保代码质量 |

### 1.3 核心类比：马与马具

**"Harness"原意是马具**——这是理解整个概念的钥匙。

```
┌─────────────────────────────────────────────────────────────┐
│                      🐴 骏马 (LLM)                          │
│                                                             │
│   天赋能力：                                                  │
│   • 跑得快 → 强大的推理能力                                  │
│   • 力量大 → 处理复杂任务                                    │
│   • 聪明 → 理解上下文和意图                                  │
│                                                             │
│   天生问题：                                                  │
│   • 没有方向感 → 不知道往哪里跑                              │
│   • 容易受惊 → 对异常情况反应不可控                          │
│   • 难以控制 → 不听指挥，想跑哪跑哪                          │
│                                                             │
│   结果：在空旷草原上跑得很快很好看，但完全没法干活            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                            ⬇️ 套上马具
┌─────────────────────────────────────────────────────────────┐
│                   🐴➕🏇 驯服的战马                          │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │                    马具系统                          │   │
│   │                                                     │   │
│   │   缰绳 ──────→ 控制方向                              │   │
│   │   │          Context Engineering                    │   │
│   │   │          告诉马往哪里走                          │   │
│   │   │                                                 │   │
│   │   马鞍 ──────→ 稳定骑乘                              │   │
│   │   │          Architectural Constraints              │   │
│   │   │          确保马不会把骑手摔下来                  │   │
│   │   │                                                 │   │
│   │   围栏 ──────→ 防止乱跑                              │   │
│   │   │          Entropy Management                     │   │
│   │   │          定期维护，防止系统衰退                  │   │
│   │   │                                                 │   │
│   │   跑道 ──────→ 明确路径                              │   │
│   │              Workflow Control                       │   │
│   │              定义任务的执行流程                      │   │
│   │                                                     │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   结果：可以拉车、耕地、送信——真正干活了！                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**深度解读**：

| 马具组件 | 功能 | Harness 对应 | 工程意义 |
|---------|------|-------------|---------|
| **缰绳** | 骑手拉左边马往左，拉右边马往右 | Context Engineering | 控制信息输入，引导方向 |
| **马鞍** | 让骑手稳稳坐在马背上 | Architectural Constraints | 系统级约束，防止失控 |
| **围栏** | 马跑不出牧场边界 | Entropy Management | 长期维护，防止衰退 |
| **马蹄铁** | 保护马蹄，增加抓地力 | Tools | 扩展能力，适应环境 |

**哲学思考**：马具不是限制马的自由，而是让马的能量能够被**有效利用**。同样，Harness 不是限制 AI 的能力，而是让 AI 的能力能够**可靠产出**。

### 2.2 支柱一：Context Engineering（上下文工程）

#### 核心问题：Agent 应该看到什么？

**哲学本质**：信息可见性决定行为边界。

```
┌─────────────────────────────────────────────────────────────┐
│                  Context Engineering                         │
│                                                             │
│   核心洞察：                                                  │
│   ─────────────────────────────────────                     │
│   从 Agent 的视角看，任何它在上下文中无法访问的东西都不存在  │
│                                                             │
│   这就像：                                                    │
│   • 你看不见的东西，你不会考虑                               │
│   • 你不知道的规则，你不会遵守                               │
│   • 你不了解的历史，你无法借鉴                               │
│                                                             │
│   ┌───────────────────────────────────────────────────┐     │
│   │              Agent 的"世界观"                      │     │
│   │                                                   │     │
│   │   Slack 里的讨论 ──→ ❌ 不可见                     │     │
│   │   人脑中的知识 ──→ ❌ 不可见                       │     │
│   │   未提交的代码 ──→ ❌ 不可见                       │     │
│   │   口头传达的规范 ──→ ❌ 不可见                     │     │
│   │                                                   │     │
│   │   对 Agent 来说，这些"不存在"                     │     │
│   └───────────────────────────────────────────────────┘     │
│                                                             │
│   解决方案：                                                  │
│   ─────────────────────────────────────                     │
│   把所有 Agent 需要知道的东西编码进代码仓库                  │
│   • 架构决策 → ADR 文档                                     │
│   • 设计文档 → /docs 目录                                   │
│   • 质量标准 → Linter 配置                                  │
│   • 执行计划 → AGENTS.md                                    │
│   全部版本化、可发现、可追溯                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 支柱二：Architectural Constraints（架构约束）

#### 核心问题：系统应该防止什么？

**哲学本质**：不是"建议遵守"，而是"无法违反"。

```
┌─────────────────────────────────────────────────────────────┐
│              Architectural Constraints                       │
│                                                             │
│   核心洞察：                                                  │
│   ─────────────────────────────────────                     │
│   光在文档里写规则，Agent 仍然可能违反                       │
│   在系统层面强制执行，则从根本上防止违规                      │
│                                                             │
│   两种约束方式对比：                                          │
│   ─────────────────────────────────────                     │
│                                                             │
│   软约束（文档/提示词）：                                     │
│   ┌───────────────────────────────────────────────────┐     │
│   │ "请在写代码时遵循以下规范..."                       │     │
│   │ "建议使用 TypeScript..."                           │     │
│   │ "最好不要超过 500 行..."                           │     │
│   │                                                   │     │
│   │ 结果：Agent 可能忽略、忘记、理解偏差               │     │
│   └───────────────────────────────────────────────────┘     │
│                                                             │
│   硬约束（系统强制）：                                        │
│   ┌───────────────────────────────────────────────────┐     │
│   │ Linter 报错 → 代码无法提交                         │     │
│   │ 类型检查失败 → 构建失败                            │     │
│   │ 测试不通过 → 无法合并                              │     │
│   │                                                   │     │
│   │ 结果：Agent 必须修复才能继续                       │     │
│   └───────────────────────────────────────────────────┘     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 🏭 生活实例：工厂安全管理

**场景**：工厂要求工人戴安全帽

| 方式 | 实现 | 效果 | 对应 Harness |
|------|------|------|-------------|
| **软约束** | 墙上贴标语"请戴安全帽" | ❌ 工人可能忘记 | 提示词/文档 |
| **硬约束** | 入口感应门，不戴安全帽无法通过 | ✅ 强制执行 | Architectural Constraints |

**更多生活实例**：

| 场景 | 软约束 | 硬约束 |
|------|--------|--------|
| **开车** | 路牌写"请系安全带" | 不系安全带车子无法启动 |
| **密码** | 提示"请设置强密码" | 系统拒绝弱密码 |
| **代码** | 文档写"请写测试" | 没有测试无法合并 PR |
| **AI** | 提示词写"请遵循规范" | Linter 报错必须修复 |

#### 🔄 反馈循环模型

```
┌─────────────────────────────────────────────────────────────┐
│                    自动修复反馈循环                          │

│   这是 Architectural Constraints 的核心机制：                │
│                                                             │
│   ┌──────────┐     ┌──────────┐     ┌──────────┐           │
│   │  Agent   │────→│  写代码  │────→│  Linter  │           │
│   │          │     │          │     │   检查   │           │
│   └──────────┘     └──────────┘     └────┬─────┘           │
│        ⬆️                                │                  │
```

### 3.2 Agent Loop 详解

#### 🔄 循环的本质

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent Loop 本质                          │
│                                                             │
│   为什么需要循环？                                            │
│   ─────────────────────────────────────                     │
│   • LLM 是无状态的：调一次 API 返回一个结果就结束            │
│   • 复杂任务需要多步：读文件 → 分析 → 写代码 → 测试 → 修复   │

│                                                             │
│   ┌─────────────────────────────────────────────────┐       │
│   │                                                 │       │
│   │    while not done:                              │       │
│   │        思考 → 决定行动 → 执行 → 观察结果        │       │
│   │                                                 │       │
│   └─────────────────────────────────────────────────┘       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.5 完整代码示例

```python
"""
Harness 核心实现示例
展示三大支柱如何在代码中体现
"""

class Harness:
    """
    一个完整的 Harness 实现
    """
    
    def __init__(self, llm, tools, config):
        self.llm = llm
        self.tools = tools
        self.config = config
        
        # ============ Architectural Constraints ============
        self.max_steps = config.get('max_steps', 10)  # 最大步数限制
        self.timeout = config.get('timeout', 300)      # 超时控制
        self.allowed_tools = config.get('allowed_tools', [])  # 工具白名单
        
    def run(self, user_query: str) -> str:
        """
        Agent Loop 主循环
        """
        # ============ Context Engineering ============
        # 构建初始上下文
        messages = [
            {"role": "system", "content": self._build_system_prompt()},
            {"role": "user", "content": user_query}
        ]
        
        # ============ Workflow Control ============
        for step in range(self.max_steps):  # Architectural Constraint: 防止无限循环
            
            # 调用 LLM
            response = self.llm.chat(messages)
            parsed = self._parse_response(response)
            
            # 检查是否完成
            if parsed["type"] == "final_answer":
                return parsed["content"]
            
            # 执行工具调用
            elif parsed["type"] == "tool_call":
                tool_name = parsed["tool"]
                tool_args = parsed["args"]
                
                # ============ Architectural Constraints ============
                # 检查工具是否在白名单中
                if tool_name not in self.allowed_tools:
                    error_msg = f"工具 {tool_name} 不在允许列表中"
                    messages.append({"role": "assistant", "content": response})
                    messages.append({"role": "tool", "content": error_msg})
                    continue
                
                # 执行工具
                try:
                    tool_result = self._execute_tool(tool_name, tool_args)
                except Exception as e:
                    tool_result = f"工具执行失败: {str(e)}"
                
                # ============ Context Engineering ============
                # 管理上下文：添加消息
                messages.append({"role": "assistant", "content": response})
                
                # ============ Entropy Management ============
                # 清理/压缩工具结果
                tool_result = self._compress_if_needed(tool_result)
                messages.append({"role": "tool", "content": tool_result})
                
                # 上下文管理：检查是否需要压缩
                if self._should_compress(messages):
                    messages = self._compress_context(messages)
        
        # ============ Architectural Constraints ============
        return "达到最大步数限制，任务未完成"
    
    def _build_system_prompt(self) -> str:
        """
        Context Engineering: 构建系统提示词
        """
        # 这里会读取 AGENTS.md 等文档
        return """
        你是一个代码助手。
        
        # 项目概览
        ...
        
        # 代码规范
        ...
        
        # 可用工具
        ...
        """
    
    def _compress_if_needed(self, content: str) -> str:
        """
        Entropy Management: 压缩过长的内容
        """
        max_length = self.config.get('max_tool_result_length', 10000)
        if len(content) > max_length:
            # 使用小模型进行摘要
            return self._summarize(content)
        return content
    
    def _should_compress(self, messages: list) -> bool:
        """
        Context Engineering: 判断是否需要压缩上下文
        """
        total_length = sum(len(m.get('content', '')) for m in messages)
        return total_length > self.config.get('context_threshold', 100000)
```

### 6.2 三种评分器

```
┌─────────────────────────────────────────────────────────────┐
│                    三种评分器类型                            │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │ 代码评分器 (Code-based Graders)                      │   │
│   │                                                     │   │
│   │ 方法：字符串匹配、单元测试、静态分析                 │   │
│   │ 优点：快速、便宜、客观、可复现                       │   │
│   │ 缺点：对有效变体脆弱                                 │   │
│   │                                                     │   │
│   │ 生活类比：选择题考试                                 │   │
│   │ • 答案非对即错                                       │   │
│   │ • 批改快速准确                                       │   │
│   │ • 但无法评估创造性                                   │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │ 模型评分器 (Model-based Graders)                     │   │
│   │                                                     │   │
│   │ 方法：基于评分标准、自然语言断言                     │   │
│   │ 优点：灵活、可扩展、捕获细微差别                     │   │
│   │ 缺点：非确定性、需要人工校准                         │   │
│   │                                                     │   │
│   │ 生活类比：作文评分                                   │   │
│   │ • 需要理解内容                                       │   │
│   │ • 有评分标准但主观                                   │   │
│   │ • 不同老师可能给不同分                               │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │ 人工评分器 (Human Graders)                           │   │
│   │                                                     │   │
│   │ 方法：专家审查、众包判断                             │   │
│   │ 优点：黄金标准、匹配专家判断                         │   │
│   │ 缺点：昂贵、慢                                       │   │
│   │                                                     │   │
│   │ 生活类比：论文答辩                                   │   │
│   │ • 专家面对面评审                                     │   │
│   │ • 最权威但成本最高                                   │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   组合使用策略：                                              │
│   ─────────────────────────────────────                     │
│   基础层：代码评分器（快速、客观）                            │
│   中间层：模型评分器（灵活、可扩展）                          │
│   顶层：人工评分器（校准、验证）                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 能力评估 vs 回归评估

```
┌─────────────────────────────────────────────────────────────┐
│              能力评估 vs 回归评估                            │
│                                                             │
│   能力评估（Capability Evals）                                │
│   ─────────────────────────────────────                     │
│   问题："这个 Agent 能做好什么？"                            │
│   特点：从低通过率开始，给团队一个攀登的目标                  │
│                                                             │
│   回归评估（Regression Evals）                                │
│   ─────────────────────────────────────                     │
│   问题："Agent 还能处理以前能做的所有任务吗？"               │
│   特点：应该有接近 100% 的通过率                             │
│                                                             │
│   生活类比：                                                  │
│   ─────────────────────────────────────                     │
│                                                             │
│   ┌─────────────────────────────────────────────────┐       │
│   │              学生考试类比                         │       │
│   │                                                 │       │
│   │   能力评估 = 学习新知识                          │       │
│   │   • 刚学新内容时，考试分数低                     │       │
│   │   • 随着学习深入，分数逐渐提高                   │       │
│   │   • 目标：掌握新技能                             │       │
│   │                                                 │       │
│   │   回归评估 = 复习旧知识                          │       │
│   │   • 已经掌握的内容，考试分数应该高               │       │
│   │   • 如果分数下降，说明忘记了                     │       │
│   │   • 目标：确保不遗忘                             │       │
│   │                                                 │       │
│   └─────────────────────────────────────────────────┘       │
│                                                             │
│   评估生命周期：                                              │
│   ─────────────────────────────────────                     │
│                                                             │
│   能力评估（低通过率）                                        │
│        │                                                     │
│        ▼ 改进 Agent                                          │
│   能力评估（高通过率）                                        │
│        │                                                     │
│        ▼ "毕业"                                              │
│   回归评估（持续运行）                                        │
│                                                             │
│   "我们能做到吗？" → "我们还能可靠地做到吗？"                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.4 处理非确定性

```
┌─────────────────────────────────────────────────────────────┐
│                   处理非确定性                               │
│                                                             │
│   问题：Agent 行为在运行间有变化                              │
│   同样的任务，可能这次成功，下次失败                         │
│                                                             │
│   两个关键指标：                                              │
│   ─────────────────────────────────────                     │
│                                                             │
│   pass@k：k 次尝试中至少一次正确的概率                       │
│   ─────────────────────────────────────                     │
│   • k 增加时，pass@k 上升                                    │
│   • 例：50% pass@1 = 第一次尝试成功一半任务                  │
│   • 例：80% pass@3 = 三次尝试至少成功一次的概率是 80%        │
│                                                             │
│   生活类比：投篮                                             │
│   • 投一次命中率 50%                                         │
│   • 投三次至少进一次的概率更高                               │
│                                                             │
│   pass^k：k 次试验全部成功的概率                             │
│   ─────────────────────────────────────                     │
│   • k 增加时，pass^k 下降                                    │
│   • 例：75% 单次成功率，3 次试验全通过概率 = 0.75³ ≈ 42%     │
│                                                             │
│   生活类比：考试                                             │
│   • 每次考试通过率 75%                                       │
│   • 连续三次考试都通过的概率只有 42%                         │
│                                                             │
│   选择指南：                                                  │
│   ─────────────────────────────────────                     │
│                                                             │
│   ┌─────────────────────────────────────────────────┐       │
│   │                                                 │       │
│   │   pass@k 适合：                                  │       │
│   │   • 工具类应用                                   │       │
│   │   • 一次成功就够了                               │       │
│   │   • 例：代码生成、问题解答                       │       │
│   │                                                 │       │
│   │   pass^k 适合：                                  │       │
│   │   • 面向用户的 Agent                             │       │
│   │   • 一致性至关重要                               │       │
│   │   • 例：客服机器人、自动驾驶                     │       │
│   │                                                 │       │
│   └─────────────────────────────────────────────────┘       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 第八章：总结——工程哲学

### 8.1 核心要点

```
┌─────────────────────────────────────────────────────────────┐
│                     核心要点                                 │
│                                                             │
│   1. Harness 是"确定性的壳"包裹"不确定性的核"                │
│      LLM 输出不确定，Harness 用确定性代码约束它              │
│                                                             │
│   2. Harness 拥有状态，LLM 没有                               │
│      通过 Harness 维护消息列表、Trace、执行步数              │
│                                                             │
│   3. Harness 定义 Agent 的"能力边界"                          │
│      能调哪些工具、最多循环几次、错误怎么处理                 │
│                                                             │
│   4. 模型质量在快速趋同，Harness 是长期竞争力                 │
│      每个团队必须自己搭建的资产                              │
│                                                             │
│   5. 评估是不可或缺的基础设施                                 │
│      没有评估，就是在黑暗中摸索                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 8.2 工程哲学总结

```
┌─────────────────────────────────────────────────────────────┐
│                   Harness 工程哲学                           │
│                                                             │
│   ┌─────────────────────────────────────────────────┐       │
│   │                                                 │       │
│   │   "约束创造自由"                                 │       │
│   │                                                 │       │
│   │   就像：                                         │       │
│   │   • 河流需要堤坝才能发电灌溉                     │       │
│   │   • 赛车需要赛道才能极速飞驰                     │       │
│   │   • 音乐需要节拍才能和谐动听                     │       │
│   │   • AI 需要 Harness 才能可靠产出                 │       │
│   │                                                 │       │
│   │   Harness 不是限制 AI 的能力，                   │       │
│   │   而是让 AI 的能力能够被有效利用。               │       │
│   │                                                 │       │
```
