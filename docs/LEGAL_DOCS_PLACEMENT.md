# 法律与对外协议文档：全产品规范落脚点

> 本文定义本仓库内**协议类内容**应放在哪里、如何流转、与前台如何对齐。适用于所有产品线（订阅、加油包、权益包、商户等）。

---

## 1. 对用户生效的「正式文本」在哪里

| 文档 | 代码位置 | 前台路由 | 说明 |
|------|----------|----------|------|
| 《用户服务协议》 | `frontend/src/pages/UserAgreementPage.tsx` | `/agreement` | 当前为页面内嵌正文，用户实际同意的是**此处展示版本**。 |
| 《隐私政策》 | `frontend/src/pages/PrivacyPolicyPage.tsx` | `/privacy` | 同上。 |

**原则**：司法与客诉口径以**线上页面当前展示内容**为准，不以 `docs/` 为准。

---

## 2. 仓库内 `docs/` 的角色

| 用途 | 路径约定 | 说明 |
|------|----------|------|
| 协议/条款**草案与摘录** | `docs/LEGAL_EXCERPT_*.md` | 供产品、法务、运营起草与评审；**不构成**自动对用户生效。 |
| 产品决策与 PRD | `docs/PRODUCT_*.md` 等 | 可引用摘录路径，但**不替代**主协议页面。 |

若仓库 `.gitignore` 忽略 `docs/`，新增文档入库时使用：

```bash
git add -f docs/LEGAL_DOCS_PLACEMENT.md
```

---

## 3. 推荐工作流（草案 → 定稿 → 上线）

1. **草案**：在 `docs/LEGAL_EXCERPT_*.md` 或与 PRD 同目录迭代，标注「草案」与修订表。  
2. **定稿**：法务确认**版本号**（如 `LEGAL-UA-1.2`）与**生效日期**。  
3. **上线**：将**同版本正文**合并入 `UserAgreementPage` 或新增独立页面（如专项条款 `/agreement/xxx`），并在 App 路由注册。  
4. **交叉链接**：购买页、结算页、勾选框链到 `/agreement` 对应锚点或专项页。  
5. **客服**：话术文档标注与协议相同的版本号，避免口径漂移。

---

## 4. 新增一类「对外条款」时的检查清单

- [ ] 是否在 `docs/` 中有摘录或 PRD 说明？  
- [ ] 是否已合并进 `UserAgreementPage`（或独立协议页）并部署？  
- [ ] 路由与导航/页脚链接是否可访问？  
- [ ] 购买/支付路径是否提示用户阅读相关条款？  
- [ ] 客服知识库是否更新且版本号一致？
